# Redfish + TanStack Vue Query + `@hey-api/openapi-ts`

A Vue 3 example app that drives a [DMTF Redfish](https://www.dmtf.org/standards/redfish) BMC
through a fully-typed client generated by `@hey-api/openapi-ts`.

The example exercises:

- **End-to-end typed SDK** generated from the upstream DMTF OpenAPI document.
- **TanStack Vue Query** caching for `ServiceRoot`, `Systems`, and `Chassis`.
- **Session-based auth** (Redfish `SessionService`) backed by Pinia.
- **fetch-based SSE** streaming via `client.sse.get()` — no `EventSource`,
  so we can attach the `X-Auth-Token` header directly.
- **SSE-driven cache invalidation** — each event's `OriginOfCondition`
  is mapped back to the affected `useQuery` hooks, which refetch
  automatically; no polling and no per-feature plumbing.
- **Vite mock plugin** — the app boots end-to-end with no real BMC.

## Getting started

```bash
pnpm install

# Generate the local-only full Redfish SDK to `src/client.full/` so your
# IDE and coding agents see every endpoint and schema. Gitignored.
pnpm openapi-ts

# Start the dev server backed by the in-process mock.
pnpm dev
# → log in with any username/password
```

To talk to a real BMC instead, point Vite at it via `VITE_BMC_URL`:

```bash
VITE_BMC_URL=https://your-bmc.example.com pnpm dev
```

## Environment variables

Two scopes:

- **Runtime — read by Vite (`pnpm dev` / `pnpm preview`).**
  Vite auto-loads `.env`, `.env.local`, `.env.[mode]`, and
  `.env.[mode].local` from the example root. Putting a value in a
  gitignored `.env.development.local` is the recommended way to set
  these for local development; the shell still works as a fallback.
  - **`VITE_BMC_URL`** — if set, the Vite dev proxy forwards every
    `/redfish/...` request to this base URL (HSTS and
    content-encoding stripped, SSE held open). If unset, the
    in-process mock plugin from `mock-redfish.ts` answers every
    request so the example boots end-to-end with no real BMC. The app
    itself always uses relative URLs in dev so that the proxy
    intercepts them; in production builds the same `VITE_BMC_URL` is
    used directly as the axios baseURL.

- **Codegen — read by the `openapi-ts` CLI (`pnpm openapi-ts*`).**
  These run as plain Node and do **not** auto-load `.env*` files. Set
  them on the command line, in the shell environment, or via a tool
  like `dotenv-cli` if you want to share `.env.development.local`
  with codegen.
  - **`REDFISH_SCOPE`** — picks which client to generate. Unset (or
    `full`) writes the entire DMTF surface to gitignored
    `src/client.full/` for IDE / agent type discovery; `scoped`
    writes only the operations listed in `SCOPED_OPERATIONS` to the
    committed `src/client/`. The `pnpm openapi-ts:scoped` script
    sets this for you.
  - **`REDFISH_OPENAPI_URL`** — OpenAPI spec URL (or local path)
    used when running codegen. Defaults to the latest published DMTF
    spec; point it at `./specs/redfish.yaml` to make codegen
    offline-reproducible.

A typical `.env.development.local` (gitignored, in the example root)
looks like:

```dotenv
VITE_BMC_URL=https://your-bmc-ip
```

## Generation pipeline

The DMTF Redfish OpenAPI document is enormous: hundreds of paths and thousands
of versioned schemas. `openapi-ts.config.ts`:

1. `parser.transforms.schemaName` collapses versioned schema names like
   `Foo_v1_2_3_Foo` into `Foo`.
2. `parser.patch.input` injects the well-known Redfish query parameters
   (`$expand`, `$select`, …) onto every Redfish `GET`, fills in dangling
   `$ref` stubs, stamps schema descriptions with their version tag, and
   un-marks `UserName`/`Password`/`Token` as `readOnly` so the generated
   `SessionWritable` actually contains the login fields.
3. `parser.patch.operations` synthesises Orval-style `operationId`s because
   most Redfish paths have none.

The runtime SSE endpoint (`/redfish/v1/EventService/SSE`) is not declared in
the spec; the example uses `client.sse.get(...)` directly with a typed URL.

### Two generation modes — separate paths

The Redfish spec is far too large to commit in full (~10k schemas, ~9 MB
of types) for an example, but during local development you usually want
the **entire** API surface available to the IDE, autocomplete, and coding
agents. To get both: the config toggles output path **and** scope based on
the `REDFISH_SCOPE` environment variable:

| Command                  | `REDFISH_SCOPE` | Output dir         | Tracked?   | Used by                            |
| ------------------------ | --------------- | ------------------ | ---------- | ---------------------------------- |
| `pnpm openapi-ts`        | unset (full)    | `src/client.full/` | gitignored | dev / IDE / agents (read-only)     |
| `pnpm openapi-ts:scoped` | `scoped`        | `src/client/`      | committed  | the app import path, the build, CI |

Because the two outputs live at **different paths**, regenerating one
never disturbs the other:

- The dev's local `src/client.full/` survives commits. No "regenerate after
  every commit" cycle.
- CI clones the repo, gets only `src/client/` (a few hundred kB), and
  builds & tests with no spec download, no `openapi-ts` invocation, no
  `schemas.gen.ts`, no top-level barrel.

**Workflow when you need a new endpoint:**

1. Browse `src/client.full/` (or grep it) to find the operation you need.
2. Add its path to `SCOPED_OPERATIONS` in `openapi-ts.config.ts`.
3. `pnpm openapi-ts:scoped` — regenerates the committed `src/client/`.
4. Use it from the app via `@/client/...`. Commit `src/client/` along
   with your source change.

> **HMR gotcha:** `pnpm openapi-ts:scoped` rewrites four `.gen.ts` files
> three times each in a few seconds (initial emit + `oxfmt` + `eslint
--fix`). Vite's transform cache can hold a stale copy of one of those
> files after such a burst, leaving newly-imported symbols looking
> `undefined` in the browser. **If imports from `@/client/...` go quiet
> after a regen, restart the dev server with `pnpm dev:fresh`** — that
> wipes `node_modules/.vite/` and starts Vite with a clean dep cache.

A CI step can guard against drift:

```bash
pnpm openapi-ts:scoped
git diff --exit-code src/client/   # fails if commit didn't include the regen
```

## Release build (`npm install` + `npm run build`)

The build pipeline is intentionally network-free after `npm install`:

```bash
rm -rf node_modules
npm --loglevel info --proxy=$http_proxy --https-proxy=$https_proxy install
npm run build
```

| Step            | Network?                       | Purpose                                                                      |
| --------------- | ------------------------------ | ---------------------------------------------------------------------------- |
| `npm install`   | yes (registry only, via proxy) | Resolve and download dependencies.                                           |
| `npm run build` | **no**                         | `vue-tsc --build --force && vite build` against the committed `src/client/`. |

Guarantees:

- No `pre*`/`post*` hooks fetch anything.
- `openapi-ts` is **only** invoked by the explicit `openapi-ts` /
  `openapi-ts:scoped` scripts, never as part of `build`.
- `@hey-api/openapi-ts` is declared with a regular semver range
  (`^0.92.4`) — not the pnpm-only `workspace:*` protocol — so
  `npm install` resolves it from the registry. (Run `pnpm install` from
  the monorepo root for the workspace-linked version.)
- The build's input is whatever is committed at `src/client/`, so the
  release artifact is whichever scope was last committed (`pnpm
openapi-ts:scoped` is the supported way to refresh that).

If your pipeline forwards extra flags (e.g. `EXTRA_OPENPM="-- --base=/foo"`
or similar), `npm run build -- <flags>` appends them to the trailing
`vite build` invocation, exactly as before.

### Optional: offline regeneration

If you want `pnpm openapi-ts:scoped` itself to be network-free in CI, vendor
the DMTF spec into the repo and point the config at it:

```bash
curl -L https://raw.githubusercontent.com/DMTF/Redfish-Publications/refs/heads/main/openapi/openapi.yaml \
  -o specs/redfish.yaml
git add specs/redfish.yaml
```

Then run `REDFISH_OPENAPI_URL=./specs/redfish.yaml pnpm openapi-ts:scoped`.
The build itself never reads the spec; only generation does.

## SSE-driven Vue Query cache invalidation

The example implements the runtime contract documented in
[`docs/designs/vue-query-sse-cache-invalidation.md`](docs/designs/vue-query-sse-cache-invalidation.md):

1. Every event landing on `/redfish/v1/EventService/SSE` flows through
   `parseSSEEvent` → SSE Pinia store → `useSSEQueryInvalidation`
   (mounted once from `App.vue`).
2. The engine extracts `OriginOfCondition` and resolves each cached
   query's `_id` to a concrete URL via the
   `REDFISH_OPERATION_URLS` registry in `sseInvalidationRules.ts`.
3. Any cached query whose URL is a parent of (or equals) the event
   URI is invalidated; `ResourceCreated` / `ResourceRemoved` also
   refresh the parent collection.
4. A static rule table catches `TaskEvent` / `Update`-registry events
   that lack a usable `OriginOfCondition`.
5. `EventBufferExceeded` is the only event that triggers a cache-wide
   `invalidateQueries({})` — the rest of the engine exists to make
   that fallback as rare as possible.

When you add a new operation to `SCOPED_OPERATIONS` in
`openapi-ts.config.ts`, also add a row to `REDFISH_OPERATION_URLS`
(operation id → URL template) so the engine can target it. Operations
without a registry entry are silently ignored, which is the right
default for mutations and lifecycle endpoints that no view actually
caches.

## Auth flow

1. `LoginView` calls `postSessionServiceSessions(...)` directly so it can
   read the `X-Auth-Token` header from the response (the generated
   `Mutation` helper unwraps to the body).
2. `auth.ts` mirrors the token into `sessionStorage`.
3. `client-setup.ts` installs an axios request interceptor that forwards the
   token on every SDK call, plus a 401 response interceptor that clears the
   session and bounces to `/login`.
4. `useSSE` injects the token into the fetch-based SSE stream via
   `onRequest` so retries always pick up the current value.
